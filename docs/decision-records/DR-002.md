# DR-002: Code Generation Design

**Status:** Accepted
**Date:** 2026-02-28

## Context

Phase 4 adds code generation from Infrahub schemas to produce typed TypeScript interfaces. Key decisions concern the generator's output format, runtime integration, and CLI design.

## Decisions

### 1. Static Interfaces Over Runtime Class Generation

**Choice:** Generate plain TypeScript interfaces and type aliases, not runtime classes.

**Rationale:**
- Interfaces are erased at compile time — zero runtime cost
- Generated code is simple, readable, and easy to audit
- No coupling between generated types and runtime node representation
- Users can still use `InfrahubNode` at runtime; interfaces add compile-time safety

**Trade-off:** Users get type checking at write-time but not runtime validation of generated types. This is acceptable because the server already validates data.

### 2. Deterministic Output

**Choice:** All generated output is deterministic — schemas sorted by kind, fields sorted by name, stable filenames.

**Rationale:**
- Enables diffing generated code across schema versions
- Makes code review of regenerated types meaningful
- Prevents noisy diffs from non-deterministic ordering
- Enables reliable snapshot testing

### 3. Three Generated Types Per Schema

For each schema kind, we generate:
1. **`FooBar`** — Full interface with all fields (for reading nodes)
2. **`FooBarCreate`** — Input type excluding read-only fields (for creating nodes)
3. **`FooBarData`** — Minimal reference type with `id`/`hfid` (for relationship references)

**Rationale:** Mirrors common API SDK patterns (response type vs input type). The reference type enables type-safe relationship building.

### 4. Typed Client Wrapper (Not Generic Methods)

**Choice:** Generate a `TypedInfrahubClient` interface with per-kind accessor objects rather than generic `client.get<T>()` methods.

**Rationale:**
- `client.device.get(...)` is more discoverable than `client.get<InfraDevice>("InfraDevice", ...)`
- No need to pass kind strings when using the typed wrapper
- Generated facade delegates to the underlying `InfrahubClient`, so all runtime behavior is preserved
- Pattern: `const typed = createTypedClient(client); await typed.device.create({ name: "r1" })`

### 5. Flat Schema JSON Format for Codegen Input

**Choice:** The codegen command accepts a flat `{ nodes: [...], generics: [...] }` JSON format, while the `schema export` CLI command flattens the namespace-organized server output.

**Rationale:**
- Simpler input format for the generator
- Users can hand-craft or filter schema JSON files
- The CLI handles the conversion; the generator stays decoupled from server API formats

### 6. Commander for CLI

**Choice:** Use `commander` for CLI argument parsing.

**Rationale:**
- Most widely used Node.js CLI framework (~100M weekly downloads)
- Declarative API for commands, options, and subcommands
- Built-in help generation and error handling
- TypeScript types included
- Lightweight (no heavy dependencies)

### 7. Separate CLI Entrypoint

**Choice:** CLI lives at `src/cli.ts` with commands in `src/cli/`, separate from the library entrypoint `src/index.ts`.

**Rationale:**
- Library consumers don't pay for CLI dependencies
- CLI can be tree-shaken out when only importing the SDK
- Separate bin entry in package.json
- Clean separation of concerns
