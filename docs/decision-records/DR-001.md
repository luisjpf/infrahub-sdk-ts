# DR-001: Overall Architecture for Infrahub TypeScript SDK

**Date**: 2026-02-27
**Status**: Accepted
**Context**: Initial architecture decisions for the TypeScript SDK, using the Python SDK as reference.

## Decision

### 1. Async-Only API (No Sync Variant)

**Python SDK**: Provides both `InfrahubClient` (async) and `InfrahubClientSync` (sync), each duplicating the full API surface.

**TypeScript SDK**: Async-only. TypeScript/Node.js does not have synchronous HTTP APIs (nor should it). All methods return Promises. This eliminates the entire class duplication pattern and simplifies the codebase by ~40%.

### 2. Native `fetch` Over Axios

**Decision**: Use the native `fetch` API (available in Node.js 18+) rather than axios.

**Rationale**:
- Zero dependencies for HTTP — `fetch` is built into modern Node.js and all browsers
- Simpler API surface that maps well to our needs
- The transport layer is behind an interface (`HttpClient`), so users can inject axios or any other client
- Reduces bundle size for browser consumers

### 3. Zod for Runtime Validation (Config Only)

**Decision**: Use Zod for configuration validation and schema parsing, replacing Python's Pydantic.

**Rationale**:
- TypeScript types are erased at runtime; Zod provides runtime validation with type inference
- Zod schemas automatically generate TypeScript types (`z.infer<typeof schema>`)
- Used sparingly — only for config validation and API response parsing, not for every internal type
- Lightweight compared to alternatives (class-validator, io-ts)

### 4. Interface-Based Dependency Injection

**Decision**: All external dependencies are injected via interfaces, not concrete classes.

**Key interfaces**:
- `HttpClient`: Transport layer abstraction
- `Logger`: Logging abstraction (console by default)

**Rationale**:
- Enables unit testing without network calls
- Users can provide custom HTTP clients (e.g., with custom auth middleware)
- No DI container needed — simple constructor injection

### 5. Dict-Based GraphQL Query Building (Mirroring Python)

**Decision**: Keep the Python SDK's dict-based GraphQL query building approach.

**Rationale**:
- The Python SDK's approach of using nested dicts to represent GraphQL queries works well
- Easy to compose, filter, and modify programmatically
- The renderer converts dicts to GraphQL strings at the boundary
- Considered and rejected: tagged template literals (less composable), AST builders (over-engineered for our use case)

### 6. Schema-Driven Node Model

**Decision**: Nodes are dynamically constructed from schema definitions at runtime, not statically typed per-node-type.

**Rationale**:
- Infrahub schemas are user-defined and change per deployment
- Static type generation is a Phase 4 feature
- Runtime dynamic access (via getter methods) provides flexibility
- Attribute access: `node.getAttribute("name").value` for type-safe access

### 7. No Class Hierarchy Duplication

**Decision**: Single implementation per concept (no Base + Async + Sync pattern).

**Python SDK**: `InfrahubNodeBase` → `InfrahubNode` (async) + `InfrahubNodeSync` (sync), repeated for every major class.

**TypeScript SDK**: Single `InfrahubNode` class. This halves the class count and eliminates synchronization bugs between async/sync variants.

### 8. Error Hierarchy Using Native Classes

**Decision**: Error types extend native `Error` class with typed properties.

**Rationale**:
- TypeScript class inheritance works well for errors
- Each error type carries structured context (url, query, variables, etc.)
- Enables `instanceof` checks for error handling
- Mirrors the Python SDK's approach, which is clean and effective

### 9. Node.js 20+ Target

**Decision**: Target Node.js 20+ (LTS) with ES2022 module output.

**Rationale**:
- Native `fetch` support (stable since Node.js 18, mature in 20+)
- Modern JS features: top-level await, private fields, structuredClone
- ESM-first with CJS compatibility via tsconfig paths
- Node 20 is the current LTS; Node 18 reaches EOL in April 2025

### 10. Vitest for Testing

**Decision**: Use vitest as the test framework.

**Rationale**:
- Native TypeScript support without separate compilation step
- Fast execution with native ESM support
- Compatible API with Jest (easy for contributors)
- Built-in mocking, coverage, and watch mode
- Integrates with the Vite ecosystem

## Consequences

- Developers must use `async/await` for all SDK operations (no callback or sync API)
- Node.js < 20 users must polyfill `fetch` or inject a custom HttpClient
- Schema-driven nodes have runtime (not compile-time) attribute access — compile-time types come in Phase 4
- The SDK has minimal dependencies: only `zod` as a production dependency
